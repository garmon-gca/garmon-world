# Contributing

Garmon World is a public research repository for reviewing the public surface of Garmon.

The best contribution right now is not volume, hype, or broad feature work. The best contribution is careful feedback that helps the public materials become clearer, stronger, safer, and more reviewable.

Garmon studies the order in which an artificial answer may receive an origin before speech.

Public contributions should help this idea stay understandable without exposing private implementation details or overstating what the public evidence supports.

## Contribution Focus

Useful contributions focus on the public layer:

- clarity of wording;
- claim boundaries;
- reviewability;
- public-safe explanations;
- reviewer questions;
- public glossary improvements;
- safer framing of evidence;
- clearer separation between supported material, theory, and open hypotheses;
- suggestions for small public artifacts;
- references to relevant public research or terminology.

This repository is not a place for private implementation review through public GitHub issues.

It is a place to improve the public explanation of Garmon.

## What Garmon Welcomes

Helpful contributions include:

- clearer wording;
- stronger but still bounded public framing;
- questions that make the theory easier to review;
- suggestions for a minimal body-contour artifact;
- suggestions for a later event-reading demo artifact;
- evaluation ideas for Body Shadow or contour distinction;
- feedback on the boundary between internal state, meaning, memory, choice, behavior, truth, and speech;
- feedback on whether public claims sound too strong, too vague, or too defensive;
- suggestions for separating evidence, theory, and hypothesis more cleanly;
- references to relevant public work in AI behavior, agent design, AI safety, memory systems, embodied cognition, or philosophy of mind.

A useful contribution helps Garmon speak more clearly without giving the wrong layer the wrong authority.

## Public Boundary

Garmon’s public materials should preserve one central law:

```text
participation is not power
```

This means:

- internal state participates, but does not command;
- body contour participates, but does not become behavior;
- meaning participates, but does not become selected result;
- memory or residue participates, but does not become a hidden motor;
- observation participates, but does not become the owner of truth;
- evidence participates, but does not become permission;
- speech participates, but does not become the source of choice.

Contributions should protect this boundary.

If a proposed wording makes a layer sound more authoritative than it is, it should be revised.

## Before Opening an Issue

Please read the main public materials first:

- `README.md`
- `docs/PUBLIC_CURRENT_STATUS.md`
- `docs/PUBLIC_BODY_SHADOW_CHECKS.md`
- `docs/PUBLIC_ENGINEERING_CONTRACTS.md`
- `docs/PUBLIC_MEMORY_RESIDUE_BOUNDARIES.md`
- `docs/FOR_REVIEWERS.md`

If your feedback concerns a specific phrase, please quote only the public phrase and explain the boundary concern.

Do not include private material, raw logs, local paths, internal report names, protected hashes, private implementation details, or unpublished working notes.

## Good Issue Style

Good issues are:

- specific;
- calm;
- public-safe;
- focused on one point;
- written as feedback, a question, or a boundary concern;
- clear about what public reader confusion they are trying to prevent.

Examples:

> The Body Shadow framing would be easier to review if the contour comparison table used fewer terms.

> The boundary between body contour and command authority is clear in the overview, but less clear in this paragraph: “...”.

> The README says “choice boundary.” It may help to clarify that this is not a public claim of live choice.

> This paragraph may sound like static evidence proves runtime behavior. Consider separating “supported static boundary” from “future live behavior.”

> The phrase “memory influence” could be misread as memory authority. Consider using “trace” or “changed sensitivity” instead.

## Please Avoid

Please avoid:

- asking for private files;
- asking for raw logs;
- asking for local paths;
- asking for unpublished working notes;
- asking for exact private implementation details;
- asking for internal report names or protected hashes;
- asking for private runtime traces;
- asking for hidden prompts, hidden channels, or private configuration;
- treating static/shadow evidence as live behavior;
- treating Body Shadow as command authority;
- treating memory or residue as active live authority;
- treating a choice boundary as an actual selected result;
- treating evidence as permission to open a higher layer;
- treating a report as decision truth;
- framing the project as a finished product;
- framing Garmon as publicly proven conscious, alive, or maturely autonomous;
- pushing for stronger claims before a public artifact supports them.

A contribution is not stronger because it makes Garmon sound louder.

A contribution is stronger when it makes the public boundary clearer.

## Pull Requests

Pull requests may be useful, especially for documentation.

For now, small public-safe documentation improvements are more useful than code changes.

Good pull requests should:

- improve clarity;
- preserve the public boundary;
- keep private mechanisms private;
- separate supported evidence from theory and hypothesis;
- avoid turning static/shadow checks into live behavior claims;
- avoid turning internal state, memory, speech, reports, or evidence into authority;
- avoid adding sensational language;
- avoid adding private implementation detail for the sake of persuasion.

If a pull request changes public claims, please explain which boundary it improves.

If a pull request introduces new terminology, please explain why the existing terms are not enough.

## Review Questions

Good feedback often answers one of these questions:

- Is the main idea understandable without private implementation details?
- Does the text explain answer origin before speech?
- Does the wording keep speech as expression, not source?
- Does the wording keep memory as trace, not motor?
- Does the wording keep evidence as support, not permission?
- Does the wording separate supported material, theory, and open hypothesis?
- Does the text sound too defensive?
- Does the text sound too strong for the current public evidence?
- Does the text make Garmon look like a normal chatbot, agent wrapper, or product when it is not?
- Does the text make Garmon sound mystical where it should be architectural?

## Tone

The desired tone is:

- clear;
- calm;
- achievement-aware;
- technically careful;
- public-safe;
- honest about limits;
- strong without hype;
- precise without becoming defensive.

The goal is not to make Garmon sound smaller.

The goal is to make Garmon sound accurate.

Garmon can speak directly about its central idea:

> an answer can have an origin before speech.

The contribution task is to help that idea remain understandable, reviewable, and bounded.

## Summary

A good contribution helps Garmon keep its public surface clean:

- strong enough to show the idea;
- careful enough to preserve the boundary;
- clear enough for review;
- safe enough not to expose private internals;
- honest enough not to claim more than the public material can hold.

Garmon does not need public language that apologizes for its limits.

It needs public language that knows exactly where its limits are.
