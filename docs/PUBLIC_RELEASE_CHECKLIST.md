# Public Surface Checklist

This checklist is for maintaining the public GitHub surface of Garmon World.

Use it whenever public documents are updated, especially when internal work is translated into public-safe language.

The goal is to keep the public surface clear, strong, reviewable, and bounded.

Garmon does not need to sound smaller than it is.

Garmon also must not claim more than the public material can hold.

Last reviewed: 2026-05-12.

## Core Public Position

The public surface should make the central idea clear:

> Garmon studies the order in which an artificial answer may receive an origin before speech.

A public reader should understand that Garmon is not only about final text generation.

The public surface should show that speech is the last visible layer, not the first source of the answer.

A compact public chain may be used:

```text
state -> body contour -> meaning -> choice boundary -> trace -> speech
```

This chain should be presented as the public research direction and public review frame, not as a claim that the full private working system is publicly open.

## Repository Surface

Check that the repository surface does the following:

- README explains the project in simple language.
- README starts from Garmon’s positive idea, not from defensive disclaimers.
- README explains answer origin before speech.
- README shows the current public-safe achievement.
- README links to current status, Body Shadow checks, engineering boundaries, memory boundaries, review questions, and artifact planning.
- README keeps the LLM framed as the speech layer, not as the whole system.
- README keeps Garmon framed as a public research preview, not as a finished product.
- The brand mark is small and calm.
- The repository description is clear, modest, and achievement-aware.
- The public surface explains the path before final speech without turning it into a claim of public live behavior.
- The public surface does not make Garmon look like a normal chatbot, agent wrapper, prompt package, or product landing page.

## Public Description

The repository description should be strong and bounded.

Good direction:

> Public research materials for Garmon: an architecture of behavior before speech, where internal state shapes meaning before final LLM wording.

Or shorter:

> Garmon studies behavior before speech: internal state -> meaning -> choice boundary -> LLM speech.

Avoid descriptions that make Garmon sound like:

- a finished agent;
- a chatbot;
- a prompt system;
- a product launch;
- a normal LLM wrapper;
- a proven conscious or living system;
- a public live runtime system.

## Public-Safe Structure

A strong public page should separate three zones:

### Supported

What the public material can currently hold.

Examples:

- body-contour distinction before speech;
- static/shadow review surfaces;
- boundary discipline;
- Body Shadow as public-safe achievement;
- closed static choice boundary chain;
- synthetic pre-selected result boundary candidate.

### Theory

The broader organizing idea.

Examples:

- answer origin before speech;
- state -> body contour -> meaning -> choice boundary -> trace -> speech;
- speech as expression layer;
- memory as trace;
- participation is not power.

### Open Hypothesis

What remains future work.

Examples:

- body-born meaning participating in choice without becoming command;
- memory becoming deeper experience while remaining trace;
- expected state across time;
- body tendency as trajectory;
- speech becoming a more precise organ of expression;
- future public artifacts and external review.

A public page becomes safer when these zones are separate.

It becomes weaker when they are mixed.

## Before First or Wider Public Sharing

When sharing the repository publicly:

- do not post it everywhere at once;
- lead with the actual public-safe achievement, not only boundaries;
- mention Body Shadow and body-contour distinction before speech;
- mention the closed static choice boundary chain only with its boundary;
- keep the contour-not-command boundary visible;
- keep the pre-selected boundary clearly synthetic and closed;
- describe Garmon as a research preview, not as a finished product;
- share it with one calm technical audience first;
- save useful feedback;
- do not answer sharp comments immediately;
- update public wording only after review;
- avoid making claims stronger in replies than they are in the repository.

Public sharing should be slow enough to preserve accuracy.

It should not be so cautious that the achievement disappears.

Do not broaden public sharing if:

- any public-safe check fails;
- README sounds too strong;
- README sounds too defensive;
- Body Shadow is described as live behavior;
- static/shadow evidence sounds like implementation permission;
- the pre-selected result boundary candidate sounds like an actual selected result;
- the artifact or demo plan sounds like a finished public system;
- private material is accidentally included;
- memory or residue sounds like live authority;
- speech sounds like the source of choice;
- reports sound like decision truth;
- evidence sounds like permission;
- Garmon sounds like a proven conscious or living system;
- Garmon sounds like an empty disclaimer project;
- Vitaliy feels rushed.

Stop conditions are not failures.

They are signals to restore the boundary before moving wider.

A good public reply should leave the reader clearer, not more impressed by noise.
