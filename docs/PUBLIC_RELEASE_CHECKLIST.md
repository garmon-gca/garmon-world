# Public Release Checklist

This checklist should be completed before switching the repository from private to public.

## Repository Surface

- README explains the project in simple language.
- README shows the current status.
- README links to boundaries, review questions, and demo planning.
- The brand mark is small and calm.
- The repository description is modest.

## Safety

- No private workbench files are included.
- No raw logs are included.
- No local machine paths are included.
- No unpublished working notes are included.
- No exact private control details are included.
- No strong claims are made.

## Public Framing

- Garmon is described as an early research prototype.
- The LLM is described as the speech layer.
- Behavior preparation is described at a high level only.
- The first demo is clearly marked as not ready yet.
- External review is invited as feedback, not as validation.

## GitHub Settings To Check

- Repository visibility is still private until the final review is complete.
- Issues are enabled only when Vitaliy is ready to receive public questions.
- Discussions are enabled only when a calm public space is needed.
- Actions are not required yet.
- Releases are not required yet.
- Packages are not required yet.

## First Public Day

When the repository becomes public:

- do not post it everywhere at once;
- share it with one calm technical audience first;
- save useful feedback;
- do not answer sharp comments immediately;
- update public wording only after review.

## Stop Conditions

Do not make the repository public if:

- any public-safe check fails;
- the README sounds too strong;
- the demo plan sounds like a finished result;
- private material is accidentally included;
- Vitaliy feels rushed.

## Current Recommendation

Keep the repository private until one final human reading pass is done from the point of view of an outside technical reader.
